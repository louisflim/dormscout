## **HomePage**

File: frontend/src/components/pages/Home/HomePage.jsx

### **Purpose/Description**
Landing page that serves as the entry point for user login. Provides a unified login interface for both tenants and landlords with account type selection.

### **Key Features**
- Dual login mode (Tenant/Landlord toggle)
- Integrated login form with email/password input
- Account type validation before navigation
- Branding with photo collage carousel
- Real-time account type detection from stored user profiles

### **User Interactions**
- Select user type (Tenant/Landlord) via button toggle
- Enter email and password credentials
- Submit login form
- View account type mismatch error if attempting wrong account type login
- Click DormScout branding to reset/scroll to top

### **API Calls/Endpoints Used**
- `AuthContext.login(email, password)` - validates credentials and user type
- Reads from `dormScoutUsers` localStorage (simulated backend)
- Checks `userType` from stored user object

### **State Management**
- **Local State:**
  - `selectedType` - currently selected user type (tenant/landlord)
  - `email` - form field
  - `password` - form field
  - `error` - login error message
  - `loading` - submission loading state
- **Context Used:** `useAuth()` for login method
- **LocalStorage Keys:** `dormScoutUsers`, `dormScoutUser`, `userType`

### **Error Handling**
- Account type mismatch validation: "This account is registered as {userType}. Please login as {userType}."
- Invalid credentials: "Invalid email or password"
- Form submission loading state during authentication

### **Dependencies**
- `AuthContext` for authentication functions
- React Router (`useNavigate`) for post-login navigation
- External images (dormpic1.jpg, dormpic2.jpg, dormpic3.webp)

### **Forms/Inputs Present**
- Email input field (required)
- Password input field (required)
- User Type selection buttons (Tenant/Landlord)
- Submit button

### **Navigation/Links**
- Redirect to `/overview` on successful login
- Internal routing via React Router

## **Map View**

File: frontend/src/components/pages/Map/Map.jsx
CSS: frontend/src/components/pages/Map/Map.css

### **Purpose/Description**
Interactive map interface for discovering and booking dorms. Displays available listings as markers, supports filtering, search, and booking workflow.

### **Key Features**
- **Interactive Leaflet Map:** Pan, zoom, click to view details
- **Listing Markers:** Orange pins for dorms, blue pins for universities
- **Advanced Filtering:**
  - Max distance from university
  - Max price range
  - School/University filter
  - Gender policy filter
- **Search Functionality:** Search by title, address, or university
- **Booking Workflow:** Multi-step booking process (info → date → confirmation)
- **Bookmarking:** Save listings for later
- **Review Ratings:** Star display for each listing
- **Real-time Updates:** Pending booking counts refresh

### **User Interactions**
- Click map to view listing details in popup
- Filter listings using sidebar filters
- Enter search query to find specific dorms
- Click "Book Now" to initiate booking flow
- Select move-in date in booking form
- Click bookmark icon to save listing
- View landlord profile from listing popup
- Edit listing (landlord only via `onEditListing` callback)

### **API Calls/Endpoints Used**
- `BookingContext.createBooking()` - submit booking
- `BookingContext.getPendingCount(listingId)` - get pending bookings for listing
- `BookingContext.subscribeToBookings()` - real-time pending count updates
- `BookingContext.notifyListingChange()` - notify when listing changes
- `AuthContext` - get current user data
- Reads from localStorage: `dormscout_listings`, `dormscout_bookmarks`, `dormscout_my_bookings`

### **State Management**
- **Local State:**
  - `listings` - array of available dorms from storage
  - `selectedListing` - currently selected listing for popup
  - `bookmarked` - boolean indicating if selected listing is bookmarked
  - `search` - search query string
  - `bookingStep` - 'info' or 'date' in booking workflow
  - `moveInDate` - selected move-in date
  - `pendingCounts` - object mapping listing IDs to pending booking counts
  - `maxDistance`, `maxPrice`, `schoolFilter`, `genderPolicyFilter` - filter values
  - `showFilters` - filters panel visibility
  - `landlordProfile` - landlord profile data from storage
- **Context Used:**
  - `useBooking()` - booking operations and subscriptions
  - `useAuth()` - user data and listing management

### **Geolocation Features**
- **University Markers:** All 15 universities marked with abbreviated labels
- **Distance Calculation:** Haversine formula-based distance from university
- **Boundary Enforcement:** Markers limited to Cebu City bounds (10.25-10.45 lat, 123.82-123.95 lng)
- **Location Search:** Nominatim reverse geocoding for address lookup

### **Filtering Logic**
- Listings filtered by: distance, price, university, gender policy, search term
- Filter combinations are AND-ed (all must match)

### **Error Handling**
- Invalid listing data handled with fallback empty state
- Missing images show map preview instead
- Distance calculation handles missing coordinates
- Reverse geocoding failures silently ignored

### **Dependencies**
- Leaflet.js for maps
- OpenStreetMap tiles for map data
- Nominatim API for reverse geocoding
- AuthContext, BookingContext
- University constants from `universities.js`

### **Map Display Information**
- **Center:** [10.3157, 123.8854] (Cebu City center)
- **Zoom Level:** 13 (default)
- **University Pin Color:** Blue (#2563EB)
- **Listing Pin Color:** Orange (#E8622E)

## **Listing Management**

### **ListingPage (Board View)**
File: frontend/src/components/pages/Listing/ListingPage.jsx
CSS: frontend/src/components/pages/Listing/ListingPage.css

**Purpose:** Landlord property listing management - create, read, update, delete listings with images and map placement.

**Key Features**
- **Multiple View Modes:** board (list), grid (cards), manage (form with map)
- **Listing Creation Form:**
  - Title, address, price, room type, available rooms
  - Gender policy selection
  - Description and tags
  - Image upload (up to 3 images)
  - Map-based location pinning
  - University selection
  - Amenities/tags input
- **Edit Existing Listings:** Modify any field and re-save
- **Delete Listings:** Remove from inventory
- **Map Integration:** Click map to select coordinates, auto-address lookup via reverse geocoding
- **Image Management:** Upload new images, preview, remove images
- **Tenant Management:** View applicants for each listing
- **Real-time Updates:** Subscribes to booking changes

**Validation Rules**
- Title required and non-empty
- Address required and non-empty
- Coordinates required (must pin on map)
- Price must be valid number > 0
- Room type required
- Available rooms required
- Gender policy required
- Description required and non-empty
- At least 1 image required (up to 3 max)

**Form Fields**
- Title (text, required)
- Address (text, required, auto-populated from geocoding)
- Price (number, required, > 0)
- Rooms (select: Single, Double, Triple, Shared)
- Available Rooms (number)
- Gender Policy (select: Both, Girls Only, Boys Only)
- Description (textarea, required)
- Tags (comma-separated text)
- University (select from dropdown)
- Images (file upload, multiple)
- Latitude/Longitude (readonly, set by map click)

**State Management**
- **Local State:**
  - `listings` - array of landlord's listings
  - `editingId` - ID of listing being edited (null if adding new)
  - `form` - form field values object
  - `imageFiles` - new image files selected
  - `previewUrls` - preview URLs for new images
  - `errors` - validation error messages object
  - `viewMode` - 'board', 'grid', or 'manage'
  - `selectedId` - currently selected listing for detail view
  - `landlordProfile` - landlord business info from storage
- **Context Used:**
  - `useAuth()` - user data, add/update/remove listing operations
  - `useBooking()` - pending booking counts, listing change notifications

**API Calls/Endpoints Used**
- `AuthContext.addListing(listing)` - create new listing
- `AuthContext.updateListing(id, listing)` - update existing listing
- `AuthContext.removeListing(id)` - delete listing
- Nominatim reverse geocoding API for address lookup
- Reads/writes to localStorage: `dormscout_listings`, `dormscout_landlord_profile`

**Storage Keys**
- `dormscout_listings` - shared listings storage for tenants to book from
- User's `listings` array in `dormScoutUser`

**Image Handling**
- FileReader API to convert images to data URLs
- Up to 3 images per listing
- Preview URLs created with `URL.createObjectURL()`
- Cleanup of object URLs on unmount

**Map Features**
- Leaflet map with university overlay markers
- Click to select coordinates
- Boundary validation (Cebu City only)
- Auto-address lookup on marker placement

**Errors That Can Occur**
- Duplicate listing IDs (handled by Date.now())
- Missing landlord profile (uses defaults)
- Image read failures (caught and logged)
- Storage quota exceeded (caught gracefully)

### **BookingPage**
File: frontend/src/components/pages/Booking/BookingPage.jsx
CSS: frontend/src/components/pages/Booking/BookingPage.css

**Purpose:** Display tenant's bookings with status, cancel functionality, and booking details.

**Key Features**
- **Merged Booking Display:** Combines confirmed bookings from shared storage with pending bookings from user profile
- **Booking Cards:** Each card shows property image/map, name, address, status, price, move-in date
- **Status Indicators:** Color-coded status badges (Pending, Confirmed, Rejected)
- **Booking Details Modal:** Full information popup
- **Cancellation:** Cancel booking with reason and move-out date
- **Map Preview:** Small map for properties with coordinates
- **Image Preview:** Show listing image if available

**Status Display**
- Confirmed/Accepted/Active: green #d1fae5 / #065f46
- Rejected: red #fee2e2 / #991b1b
- Pending: yellow #fef9c3 / #92400e

**Cancel Workflow**
1. Click booking card to select
2. Open cancellation modal
3. Enter reason (optional) and move-out date (optional)
4. Confirm cancellation
5. Booking removed from displays
6. Updates synced to multiple storage locations

**Data Merging Logic**
- **From 'bookings' storage:** Confirmed/rejected bookings shared between users
- **From user.bookings:** Pending bookings in user profile
- **Deduplication:** By booking ID to prevent duplicates
- **Sorting:** By booked date (newest first)

**State Management**
- `bookings` - merged array of bookings
- `selectedBooking` - currently selected booking for detail modal
- `cancelModal` - cancellation modal visibility
- `cancelReason`, `cancelMoveOutDate` - cancellation form fields

**API Calls/Storage**
- Reads from localStorage keys: `bookings`, `dormscout_bookings`, `dormScoutUser`
- `useAuth().cancelBooking(bookingId)` - cancel from AuthContext
- `useBooking().cancelBooking(bookingId, reason, moveOutDate)` - cancel from BookingContext
- Writes to: `bookings`, `dormscout_bookings`, `dormscout_my_bookings`

**Real-time Updates**
- Listens to storage changes on keys: `bookings`, `dormscout_bookings`, `dormScoutUser`
- Refresh when external updates detected

**Form Fields for Cancellation**
- Cancellation Reason (textarea, optional)
- Move-Out Date (date input, optional)

**Dark Mode Support**
- Theme colors configurable via props
- Card backgrounds, text colors, borders adapt

**Empty State**
- Shows message: "You have no active bookings"
- Button to navigate to Map View

## **Messaging**

File: frontend/src/components/pages/Messaging/Messaging.jsx
CSS: frontend/src/components/pages/Messaging/Messaging.css

### **Purpose/Description**
Direct messaging system for communication between tenants and landlords about bookings and properties.

### **Key Features**
- **Conversation List:** Display all active conversations
- **Message Thread:** Full message history with timestamps
- **Real-time Updates:** Message delivery tracking (sent, delivered, read status)
- **Typing Indicators:** Show when other user is typing
- **Message Reactions:** Quick emoji responses (not fully implemented)
- **Conversation Deduplication:** Prevents duplicate conversations with same user
- **Avatar Generation:** Color-coded initials for users
- **Desktop Notifications:** Request permission and notify on new messages
- **User Status:** Shows online/offline status with green dot

### **Message Statuses**
- **sent** - ✓
- **delivered** - ✓✓
- **read** - ✓✓ (highlighted in primary color)

### **Storage Structure**
```
conversations: { tenant: {...}, landlord: {...} }
conversations[role][convId] = {
  convId, name, avatar, online, typing, lastMessage, lastMessageTime, 
  landlordId, timestamp, unreadCount
}

messages: { tenant: {...}, landlord: {...} }
messages[role][convId] = [{
  id, from, to, text, timestamp, status, messageType
}]
```

### **Storage Keys**
- `messaging_conversations` - all conversations by role
- `messaging_messages` - all messages by role and conversation

### **Data Cleaning Operations**
- **Placeholder Removal:** Removes hardcoded placeholder conversations (Maria Santos, Juan dela Cruz, etc.)
- **Conversation Deduplication:** Groups by landlordId, keeps most recent if duplicates exist
- **Junk Cleanup:** If >100 conversations detected, wipes entire role (indicates old bug data)

### **User Interactions**
- Click conversation to open message thread
- Type and send message
- View message history with timestamps
- Click user avatar for profile/info
- Archive conversation (not implemented)
- Search conversations (not implemented in current version)
- Desktop notifications for new messages

### **State Management**
- **Local State:**
  - `conversations` - active conversation list
  - `selectedConvId` - currently open conversation ID
  - `messageText` - draft message input
  - `isTyping` - typing indicator status
  - `filteredConversations` - search/filtered results
  - `unreadCounts` - unread message counts per conversation
- **Context:** `useAuth()` for user data
- **Module-level Listeners:** `messagingListeners` array for real-time updates

### **API Calls/Endpoints Used**
- No external API calls (all localStorage-based)
- Storage operations via custom Storage helper object

### **Features NOT YET IMPLEMENTED**
- Message reactions
- Typing indicators (state prepared but not enforced)
- Message search
- Conversation archiving
- Voice/video calls
- File sharing

### **Error Handling**
- Safe JSON parsing with fallbacks
- Storage quota exceeded handled gracefully
- Invalid message data ignored
- Placeholder conversations safely removed

### **Dependencies**
- AuthContext for user information
- Lucide React for icons
- Custom avatar component with color coding

### **Notification Permissions**
- Requests desktop notification permission on component mount
- Shows OS notification for new messages
- Auto-closes notification after 5 seconds

### **Purpose/Description**
Centralized notification center for all platform events - booking updates, messages, reviews, system notifications.

### **Key Features**
- **Notification List:** Displays all notifications in reverse chronological order
- **Unread Badge:** Shows count of unread notifications in header
- **Notification Types:**
  - new_booking (📦) - Tenant receives booking confirmation
  - booking_accepted (✅) - Landlord accepts tenant application
  - booking_rejected (❌) - Landlord rejects tenant application
  - booking_cancelled (❌) - Booking cancelled
  - new_message (💬) - New message received
  - tenant_removed (🚫) - Tenant removed from property
- **Mark as Read:** Click unread notification to mark read
- **Delete Individual:** Remove notification with X button
- **Clear All:** Bulk delete all notifications
- **Real-time Updates:** Subscribes to notification changes via BookingContext

### **Notification Object Structure**
```
{
  id,
  type: 'new_booking' | 'booking_accepted' | 'booking_rejected' | 'booking_cancelled' | 'new_message' | 'tenant_removed',
  title,
  message,
  read: boolean,
  createdAt: ISO timestamp,
  relatedId, // booking ID, message ID, etc.
  icon, // emoji based on type
}
```

### **Timestamp Formatting**
- < 1 minute: "Just now"
- < 1 hour: "{minutes}m ago"
- < 24 hours: "{hours}h ago"
- 1 day ago: "yesterday"
- < 7 days: "{days}d ago"
- Otherwise: "Mon Jan 15" format

### **User Interactions**
- Click unread notification to mark as read (visual feedback)
- Click X button to delete individual notification
- Click "Clear All" button to delete all notifications
- Notifications update in real-time when events occur

### **State Management**
- **Local State:**
  - `notifications` - array of all notifications for user type
- **Subscriptions:** 
  - Subscribes to BookingContext notification changes
  - Re-fetches on component mount and when userType changes
- **Context Used:** `useBooking()`
  - `getNotifications(userType)` - fetch notifications
  - `markNotificationRead(notificationId)` - mark read
  - `deleteNotification(notificationId)` - delete single
  - `clearAllNotifications(userType)` - delete all
  - `subscribeToNotifications(callback)` - real-time updates

### **API Calls/Endpoints Used**
- All operations via BookingContext (localStorage-backed)

### **Empty State**
- Shows bell icon (🔕)
- Message: "No notifications yet."

### **Dark Mode Support**
- Theme-aware colors for background, text, borders
- Unread notification highlighting with primary color border

### **Styling Details**
- Unread notifications: white background with orange border
- Read notifications: light gray background with subtle border
- Hover effects on delete and clear buttons
- Icon emojis for quick visual scanning

### **Purpose/Description**

### **Key Features**

- **Write Review:** Submit new review with rating, tags, and detailed feedback
- **Star Rating:** 5-star interactive rating system

### **Available Dorms** (Hardcoded for current version)

2. Green Leaf Dormitory
3. BlueSky Residences

### **Review Object Structure**
```
{
  id,
  dormId,
  author,
  avatar, // initials
  date: 'Month Day, Year' format,
  rating: 1-5,
  tags: [], // array of tag strings
  body, // review text,
  helpful: number,
  userMarkedHelpful: boolean
```

## **Common Error Scenarios for Testing**

1. **Authentication:**
   - Login with wrong password → Error message
   - Login with non-existent email → Error message
   - Login with wrong user type → Error message
   - Register with duplicate email → Error message

2. **Bookings:**
   - Book without selecting move-in date → Validation error
   - Cancel booking with missing reason → Should still work (optional)
   - View bookings when none exist → Empty state

3. **Listings:**
   - Create listing without image → Validation error
   - Pinning location outside Cebu City → Alert
   - Update listing with invalid price → Validation error
   - Delete listing with active tenants → Should cascade

4. **Messaging:**
   - Send message to non-existent user → Conversation created
   - Message sent when offline → Status remains "sent"
   - Receive duplicate conversations → Auto-merged/deduped

5. **Storage:**
   - localStorage quota exceeded → Graceful failure (console error)
   - Corrupt JSON in storage → Safe fallback to empty array
   - Cross-tab sync failure → May show stale data

## **Performance Considerations**

- Listings filtered/searched on client-side (no backend search)
- Distance calculations use Haversine formula (relatively fast)
- Map re-renders only when markers change
- Image data URLs stored in localStorage (large payload)
- Notification subscription listeners may accumulate (unsubscribe on unmount)
